# 📌 환경설정
* 환경설정 후 각 파일의 readme 한번씩 읽어보면 좋을 것 같습니다. 정리가 완전히 안되어 있긴 합니다.

## 1. 로컬환경
* Github clone 후 .env 파일 추가
* 기본 세팅 -> venv 가상 환경 세팅

### BN(백엔드 서버 띄우기)
0. root 디렉토리로 이동
1. cd server
2. pip install -r requirements.txt
4. taskkill (필요시)
5. uvicorn main:app --reload

### FN(프론트엔드 서버 띄우기)
0. npm 설치(https://nodejs.org/ko/download)
1. cd client
2. (.venv) pip install npm 또는 pnpm install
3. 환경: powershell / 위치: ~/client 내부에서 pnpm install (필요시)
4. npm run dev


## 2. docker 환경
1. docker desktop 켜기
2. finalproject_backup 파일에서
3. docker compose up --build -d


1. docker compose down

# 📌 커밋 컨벤션 (Commit Convention)

## 💡 핵심 원칙
* **일관성 유지:** 커밋은 최대한 자주 작성합니다 (기능 단위, 파일 및 코드 추가/수정/삭제).
* **1 커밋 1 이슈:** 하나의 커밋에는 하나의 문제만 다룹니다. (기능 복구 및 이력 추적 용이성 확보)

## 🔤 영어 컨벤션 (Prefix)
| 태그 | 설명 | 사용 시점 |
|:---:|:---|:---|
| **Feat** | 기능 추가 | 브랜치 첫 커밋, 새로운 기능 구현 시 |
| **Add** | 파일/문서 추가 | 새로운 코드 파일이나 문서 파일 자체를 추가할 때 |
| **Modify** | 코드 수정 | 기능 수정, 버그 수정 등 개발 과정의 코드 변경 |
| **Update** | 문서/데이터 수정 | 코드 외적인 문서나 데이터 내용을 수정할 때 |
| **Delete** | 삭제 | 코드 라인 삭제 또는 파일 자체를 삭제할 때 |
| **Test** | 테스트 | 테스트 코드 추가/수정, 연동 테스트 작업 |
| **Chore** | 기타 작업 | 설정 변경 등 기능/서비스 구조에 영향이 없는 작업 |

<br>

## ⚠️ 작성 규칙 (Guidelines)
1.  **제목:** 50글자 이내로, 마침표(`.`) 없이 작성합니다.
2.  **문법:** 명령문 형태를 사용합니다 (과거형 X).
    * *Good:* "기능을 추가"
    * *Bad:* "기능을 추가했음"
3.  **내용:** '어떻게'보다는 **'무엇을', '왜'** 변경했는지 설명합니다.
4.  **본문:** 제목으로 설명이 부족할 경우, 한 줄 띄우고 본문에 상세 내용을 작성합니다.
5.  **단위:** 한 개의 커밋에는 한 개의 기능/변경사항만 작성합니다 (1커밋 1기능).
